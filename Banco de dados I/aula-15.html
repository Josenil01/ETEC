Aula 15: A Natureza dos Dados

Tipos de Dados e o Mistério do NULL: Aprenda a definir o formato da realidade, impor regras e lidar com o vazio absoluto.

No mundo físico, você não guarda leite em uma peneira nem pregos em uma caixa de vidro frágil. No banco de dados, a lógica é a mesma: cada informação precisa de um Tipo de Dado específico para ser armazenada com eficiência e segurança.

"O tipo de dado é o contrato que garante que o servidor saiba exatamente como processar aquela informação."

1. A Matrix de Tipos SQL

Numéricos

  • SERIAL Auto-incremento

    Gera valores automáticos (1, 2, 3...). Usado exaustivamente para criar Chaves Primárias (PK).

  • INT / INTEGER Números Inteiros

    Ex: 1, 42, -500. Ideal para quantidades e idades.

  • DECIMAL(p,s) Precisão Exata

    Ex: 150.50. O 'p' é o total de dígitos, o 's' são as casas decimais. Essencial para dinheiro.

Alfanuméricos

  • VARCHAR(n) Tamanho Variável

    Ex: 'João Silva'. O 'n' é o limite máximo de caracteres.

  • TEXT Tamanho Ilimitado

    Ideal para descrições longas, posts de blog ou comentários.

Cronológicos

  • DATE Apenas Data

    Formato: 'YYYY-MM-DD'. Ex: '2025-12-25'.

  • TIMESTAMP Data + Hora

    Ex: '2025-12-25 14:30:00'. Para registros de log e criação.

Lógicos

  • BOOLEAN Verdade ou Mentira

    Valores: TRUE ou FALSE. Ideal para status (ativo, pago, enviado).

2. O Vazio Absoluto: O Conceito de NULL

Um dos erros mais comuns de iniciantes é confundir NULL com zero ou vazio. O NULL não é um valor; é a ausência de informação ou um "não sei".

0
Número Zero

Eu contei e o resultado é nada.

""
String Vazia

Eu digitei, mas não escrevi nada.

NULL
O Grande Vazio

Ninguém sabe. Ninguém informou.

Regra de Ouro do NULL

Você nunca compara algo com NULL usando o sinal de igual (=). Como o NULL é desconhecido, NULL = NULL resulta em falso! Para encontrar vazios, usamos IS NULL.

3. As Regras do Contrato (Constraints)

Apenas escolher o tipo de dado não é suficiente. Precisamos impor Restrições (Constraints) para garantir a qualidade e a veracidade dos dados. Sem elas, o banco aceitaria idades negativas ou CPFs repetidos.

NOT NULL

Impede que o campo fique vazio (NULL). Obrigatório para informações vitais como 'nome' ou 'cpf'.

UNIQUE

Garante que não existam dois valores iguais nesta coluna. Ex: O 'email' de cada usuário deve ser único.

DEFAULT valor

Define um valor padrão caso nada seja enviado. Ex: DEFAULT 0 ou DEFAULT TRUE.

CHECK (condição)

Cria uma regra de validação customizada. Ex: CHECK (idade >= 18).

PRIMARY KEY & FOREIGN KEY

As restrições supremas. A PK é a combinação implícita de UNIQUE + NOT NULL. A FK obriga que o valor exista previamente na tabela de origem.

4. Laboratório: Arquitetando Tipos

Revisitando o sistema de Gestão de Projetos da Aula 08. Sua missão é definir o tipo de dado SQL e se a coluna permite ou não valores nulos.

schema_projetos.sql
CREATE TABLE tb_projeto (
    id_projeto SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL, -- Nome é obrigatório!
    data_inicio DATE NOT NULL,
    data_fim DATE, -- Pode ser NULL (o projeto pode não ter acabado)
    orcamento DECIMAL(12,2) DEFAULT 0,
    esta_ativo BOOLEAN DEFAULT TRUE
);

CREATE TABLE tb_usuario (
    id_usuario SERIAL PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    email VARCHAR(100) NOT NULL
);

CREATE TABLE tb_postagem (
    id_postagem SERIAL PRIMARY KEY,
    titulo VARCHAR(150) NOT NULL,
    conteudo TEXT, -- Pode ser vazio (NULL) se for apenas um rascunho de título
    id_usuario INT REFERENCES tb_usuario(id_usuario)
);
