Módulo 2: Modelagem Relacional Arquiteto de Banco de Dados

Aula 12: A Cura do Caos (1FN)

Introdução à Normalização: Aprenda a eliminar anomalias e a estruturar dados com a Primeira Forma Normal.

1. O Problema: Anomalias de Dados

Antes de falarmos de Normalização, precisamos de entender o porquê de ela existir. Quando criamos tabelas que parecem "folhas de cálculo" gigantescas sem regras, o banco de dados sofre de Anomalias.

Anomalia de Inserção

Não conseguimos adicionar um dado sem termos outro. Ex: Não consigo cadastrar um novo Fornecedor se ele ainda não me tiver vendido nenhum Produto (porque a tabela exige a preenchimento do Pedido).

Anomalia de Atualização

O mesmo dado está repetido em várias linhas. Ex: O "João" mudou de celular. Como o nome dele está repetido em 50 pedidos diferentes, temos de atualizar 50 linhas, arriscando-nos a esquecer alguma.

Anomalia de Exclusão

Ao apagarmos um dado, perdemos informações importantes sem querer. Ex: Ao apagar o único pedido que envolvia o "Monitor LG", apagamos acidentalmente todos os dados do fornecedor desse monitor da base de dados.

2. A Primeira Forma Normal (1FN)

A Normalização é um processo passo a passo. O primeiro passo terapêutico para curar uma tabela doente é aplicar a 1ª Forma Normal. Uma tabela está na 1FN se, e só se, obedecer a estas regras sagradas:

  • 1

    Valores Atômicos: Cada célula (interseção de linha e coluna) deve conter apenas UM ÚNICO VALOR indivisível. Sem listas, sem separações por vírgulas.

  • 2

    Sem Grupos Repetitivos: Não podemos ter colunas que repetem o mesmo tipo de dado (ex: telefone1, telefone2, telefone3).

  • 3

    Possui Chave Primária: Deve existir um identificador único para cada registro (linha) na tabela.

Objetivo principal: Transformar matrizes bidimensionais em relações matemáticas puras.

Regras Adicionais Cruciais da 1FN

  • Tratamento de Atributos Compostos: Um atributo como 'Endereço Completo' (Rua X, nº 10, Lisboa) viola a atomicidade. Na 1FN, este atributo não gera uma tabela nova, mas desdobra-se em múltiplas colunas (ex: Rua, Numero, Cidade) na mesma tabela.
  • Chaves Primárias Compostas: Quando movemos um atributo multivalorado (ex: Múltiplos Telefones) para uma nova tabela, essa tabela "filha" precisa de um identificador forte. Usamos uma Chave Primária Composta (a combinação da Chave Estrangeira com o próprio valor do dado, ex: PRIMARY KEY (id_cliente, numero_telefone)), garantindo que a combinação da FK com o dado seja única.

3. A Implementação SQL

Ao criar tabelas em SQL, a própria estrutura de dados impõe a 1FN. O PostgreSQL não possui nativamente um tipo de dados "lista" ou "array" tradicional para os comandos base, obrigando o arquiteto a desenhar chaves estrangeiras apropriadas.

schema_1fn.sql
-- INCORRETO (Não use isto):
-- CREATE TABLE pedido_errado (id INT, telefones VARCHAR(255), produtos VARCHAR(255));

-- CORRETO (1FN Aplicada - Valores Atômicos):
CREATE TABLE pedido (
    id_pedido SERIAL PRIMARY KEY,
    cliente_nome VARCHAR(100)
);

CREATE TABLE telefone (
    numero VARCHAR(20) PRIMARY KEY,
    id_pedido INT REFERENCES pedido(id_pedido)
);

CREATE TABLE pedido_produto (
    id_pedido INT REFERENCES pedido(id_pedido),
    nome_produto VARCHAR(100)
);
Aviso do Arquiteto: A 1FN resolve o problema de listas e vírgulas, mas note que o "nome do produto" (ex: Mouse RGB) começou a repetir-se várias vezes na tabela pedido_produto. Essa redundância textual causa Anomalias de Atualização, que serão curadas na nossa próxima etapa: a 2ª e 3ª Formas Normais!

Tempo: 5 min

Neste sistema de uma escola de programação, o programador tentou evitar usar vírgulas criando várias colunas para o mesmo fim. Aplique a 1FN.

Desafio Bônus: O Registro de Clientes VIP

Tempo: 10 min

Este sistema guarda o registro dos clientes VIP da nossa loja. No entanto, o programador cometeu dois erros graves que violam a 1FN: agrupou todo o endereço numa só célula e permitiu múltiplas redes sociais no mesmo campo. Aplique a 1FN.

Solução:

1) O "Endereço Completo" (Atributo Composto) é desdobrado em várias colunas na própria tabela.
2) As "Redes Sociais" (Multivalorado) vão para uma nova tabela com uma Chave Primária Composta.

PK ID_VIP

Username
PRIMARY KEY (ID_VIP, Username)
