Módulo 3: Infraestrutura e Nuvem Arquiteto de Banco de Dados

Aula 16: O Gênesis dos Dados

DDL (Data Definition Language): Aprenda a materializar as suas estruturas na nuvem criando bancos e tabelas base no Neon.

A linguagem SQL (Structured Query Language) é dividida em várias "famílias" de comandos. Hoje conheceremos a família mais importante para um Arquiteto de Software: a DDL (Data Definition Language).

Com a DDL, não manipulamos os dados (não inserimos, nem lemos registros). Nós construímos a casa onde os dados vão morar. É aqui que definimos as tabelas, as colunas e as regras que estudamos na aula passada.

1. CREATE DATABASE: O Terreno

Antes de ter tabelas, você precisa de um banco de dados (o contêiner principal). O comando para isso é extremamente simples:

CREATE DATABASE empresa_db;

A Nuvem faz isso por você!

Em plataformas modernas como o Neon.tech, quando você clica em "New Project" no Dashboard, o servidor roda esse comando automaticamente por trás dos panos. Por isso, no Neon, seu banco padrão já nasce criado com o nome neondb.

2. CREATE TABLE: A Planta da Casa

A anatomia de um comando de criação de tabela une tudo o que vimos até hoje: o nome da coluna, o tipo de dado e a restrição (constraint). A sintaxe padrão segue esta regra:

Sintaxe Padrão
CREATE TABLE nome_da_tabela (
    nome_coluna_1 TIPO_DE_DADO CONSTRAINT,
    nome_coluna_2 TIPO_DE_DADO,
    nome_coluna_3 TIPO_DE_DADO CONSTRAINT
);

As Regras da Casa (Constraints)

PRIMARY KEY

O identificador único e absoluto da linha. Combina automaticamente as regras NOT NULL e UNIQUE.

NOT NULL

Proíbe o vazio. Impede que o registro seja salvo se este campo não for preenchido (ex: nome, cpf).

UNIQUE

Garante exclusividade. O banco não aceitará dois registros com o mesmo valor nesta coluna (ex: email).

DEFAULT valor

Salva a pátria preenchendo o campo com um valor padrão caso o usuário não envie nada (ex: DEFAULT 0).

CHECK (condição)

Cria uma regra de negócio customizada diretamente no banco (ex: CHECK (idade >= 18)).

REFERENCES

Cria a Chave Estrangeira (FK). Obriga que o valor inserido já exista na tabela de origem apontada.

Atenção à Vírgula!

O erro mais comum ao programar SQL é esquecer a vírgula no final de cada linha de coluna, ou colocar uma vírgula na última coluna antes do fecho dos parênteses. A última coluna nunca tem vírgula.

3. Laboratório: Código na Nuvem

Duração: 40 min

Sua missão agora é acionar o Editor SQL do seu projeto no Neon.tech e materializar duas tabelas essenciais para o sistema corporativo: Departamento e Funcionário.

Nota: Não criaremos a Chave Estrangeira (FK) ainda! Focaremos apenas nas Tabelas Base.

Tabela 1: DEPARTAMENTO

Entidade forte, não depende de ninguém.

  • id_depto (Auto-incremento e Chave Primária)
  • nome (Texto até 100 caracteres, Obrigatório)
  • localizacao (Texto até 50 caracteres)
neon_editor.sql
CREATE TABLE departamento (
    id_depto SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    localizacao VARCHAR(50)
);

Tabela 2: FUNCIONÁRIO

Tabela base de funcionários (sem ligar ao depto ainda).

  • id_func (Auto-incremento e Chave Primária)
  • nome (Texto longo, Obrigatório)
  • cargo (Texto curto)
  • salario (Monetário)
  • data_contratacao (Apenas Data)
neon_editor.sql
CREATE TABLE funcionario (
    id_func SERIAL PRIMARY KEY,
    nome VARCHAR(150) NOT NULL,
    cargo VARCHAR(50),
    salario DECIMAL(10,2),
    data_contratacao DATE
);

5. Desafios do Arquiteto (Prática)

Fixação

No mundo real, os clientes não ditam colunas SQL; eles contam histórias. Leia os relatos abaixo, interprete as regras de negócio e use o simulador para forjar as tabelas.

Desafio 1: O Requisito do App

A equipe de produto do novo aplicativo "Connecta" enviou a seguinte User Story (História de Usuário):

"Como usuário, quero me registrar no aplicativo informando meu nome completo e meu e-mail, para poder acessar a plataforma. O sistema deve garantir de forma absoluta que não podemos ter contas em branco e que ninguém mais use o meu e-mail. Além disso, precisamos registrar a data exata em que a conta foi criada."

Sua missão: Sabendo que o banco de dados sempre precisa identificar unicamente cada registro de forma sequencial (ID), deduza a estrutura, aplique as constraints corretas e modele a tabela usuario no simulador.

solucao_usuario.sql
CREATE TABLE usuario (
    id_usuario SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL,
    email VARCHAR(150) NOT NULL UNIQUE,
    data_cadastro DATE
);

Desafio 2: O Gerente de Estoque

Você foi chamado para estruturar o inventário de uma loja online. Na reunião de alinhamento, o gerente de estoque foi categórico:

"Todo produto que entra no nosso sistema precisa de uma identificação numérica automática e um nome (não podemos de jeito nenhum ter produtos anônimos!). Além disso, precisamos guardar o valor de venda (com precisão para centavos). Para facilitar a vida de quem digita, a nossa regra de ouro é: se cadastrarmos um produto e não falarmos nada sobre o status do estoque, o sistema deve assumir por conta própria que ele está disponível (verdadeiro)."

Sua missão: Traduza este relato corporativo para o código DDL, modelando a tabela produto.

solucao_produto.sql
CREATE TABLE produto (
    id_produto SERIAL PRIMARY KEY,
    nome VARCHAR(150) NOT NULL,
    preco DECIMAL(10,2),
    em_estoque BOOLEAN DEFAULT TRUE
);

Desafio Extra: O Mini Mundo

A startup "TechShop" contratou-o como Arquiteto de Dados. O CEO explicou o negócio através de um áudio no WhatsApp:

"Fala, mestre! Seguinte, o nosso app tem Clientes (só precisamos guardar o nome deles e gerar um ID). Nós vendemos Produtos (que também têm nome), mas para a loja ficar organizada, cada produto pertence obrigatoriamente a uma Categoria (ex: Eletrônicos, Móveis). E aí vem a mágica: o Pedido! Um pedido acontece numa data específica, e cruza a informação de qual cliente comprou qual produto."

Sua missão: Mapeie as 4 tabelas descritas no áudio. Deduza as chaves primárias (PK) e desenhe as conexões utilizando as Chaves Estrangeiras (REFERENCES) apropriadas para ver o simulador desenhar as linhas de relacionamento!

solucao_minimundo.sql
CREATE TABLE cliente (
    id_cliente SERIAL PRIMARY KEY,
    nome VARCHAR(100) NOT NULL
);

CREATE TABLE categoria (
    id_categoria SERIAL PRIMARY KEY,
    nome_categoria VARCHAR(50) NOT NULL
);

CREATE TABLE produto (
    id_produto SERIAL PRIMARY KEY,
    nome VARCHAR(150) NOT NULL,
    id_categoria INT REFERENCES categoria(id_categoria)
);

CREATE TABLE pedido (
    id_pedido SERIAL PRIMARY KEY,
    data_compra DATE,
    id_cliente INT REFERENCES cliente(id_cliente),
    id_produto INT REFERENCES produto(id_produto)
);
