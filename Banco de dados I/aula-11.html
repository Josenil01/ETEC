Chegou a hora de provar o seu valor. Do rascunho em papel até à implementação SQL final, complete os 5 níveis de modelação.

Relembrando a Aula Passada

1. A Base

  • Entidade vira Tabela.
  • Atributo vira Coluna.
  • Identificador vira PRIMARY KEY.

2. Regra 1:N

A Chave Estrangeira (FK) viaja sempre para a tabela que representa o lado "N" (Muitos) do relacionamento.

3. Regra N:N

Gera uma nova Tabela Associativa. Ela absorve as chaves primárias das duas entidades originais como FKs.

Exercício 1: Aquecimento DDL

Tempo estimado: 10 min

Cenário: Estamos a criar um sistema para uma Clínica. O nível lógico já foi definido para si. Precisamos apenas do código SQL para criar a tabela de Pacientes.

Tabela: paciente
- id_paciente (Chave Primária, Autoincremento)
- nome (Texto até 100 caracteres)
- data_nascimento (Data)

Escreva o comando CREATE TABLE correspondente abaixo para visualizar a matriz:

solucao.sql
CREATE TABLE paciente (
    id_paciente SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    data_nascimento DATE
);

Exercício 2: A Propagação da Chave (1:N)

Tempo estimado: 20 min

Cenário: Na mesma Clínica, os pacientes ficam internados em quartos. A regra de negócio diz: "Um Quarto pode abrigar Vários Pacientes (1:N), mas um Paciente só pode estar num único Quarto por vez".

QUARTO (id_quarto, numero)
--- 1 : N ---
PACIENTE (id_paciente, nome)

Tarefa: Escreva o código SQL para criar as duas tabelas. Lembre-se de criar a Chave Estrangeira (FK) na tabela correta usando REFERENCES.

solucao.sql
-- 1º: Criamos a tabela forte (lado 1) que não tem dependências
CREATE TABLE quarto (
    id_quarto SERIAL PRIMARY KEY,
    numero INT
);

-- 2º: Criamos a tabela dependente (lado N) recebendo a FK
CREATE TABLE paciente (
    id_paciente SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    id_quarto INT REFERENCES quarto(id_quarto)
);

Exercício 3: O Elo Associativo (N:N)

Tempo estimado: 25 min

Cenário: Um MÉDICO realiza consultas com vários PACIENTES, e um PACIENTE pode ser consultado por vários MÉDICOS. O losango do relacionamento chama-se "Consulta" e possui os atributos data_hora e diagnostico.

Tarefa: Converta este modelo para o relacional e escreva as três tabelas necessárias em SQL no editor abaixo.

solucao.sql
CREATE TABLE medico (
    id_medico SERIAL PRIMARY KEY,
    crm VARCHAR(20),
    nome VARCHAR(100)
);

CREATE TABLE paciente (
    id_paciente SERIAL PRIMARY KEY,
    nome VARCHAR(100)
);

-- Tabela Associativa nascida do N:N
CREATE TABLE consulta (
    id_consulta SERIAL PRIMARY KEY, -- PK própria recomendada
    id_medico INT REFERENCES medico(id_medico),
    id_paciente INT REFERENCES paciente(id_paciente),
    data_hora TIMESTAMP,
    diagnostico TEXT
);

Exercício 4: Leitura de Requisitos

Tempo estimado: 30 min

O Cliente solicitou: "Preciso de um sistema para a minha biblioteca. Nós temos Autores (com nome e nacionalidade) e Livros (com título e ano_publicacao). Um livro pode ser escrito por vários autores, e um autor escreve vários livros. Ah! Também precisamos de registar as Editoras (nome e cnpj). Uma editora publica vários livros, mas cada edição de um livro tem apenas uma editora."

Tarefa: Identifique mentalmente as cardinalidades (1:N, N:N) deste texto. De seguida, escreva todo o código SQL para gerar esta estrutura.

Dica: Cuidado com a ordem em que cria as tabelas! Tabelas com FKs devem ser criadas depois das tabelas base.

solucao.sql
-- Ordem importa: Editora e Autor não dependem de ninguém.
CREATE TABLE editora (
    id_editora SERIAL PRIMARY KEY,
    cnpj VARCHAR(14),
    nome VARCHAR(100)
);

CREATE TABLE autor (
    id_autor SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    nacionalidade VARCHAR(50)
);

-- Livro depende da Editora (Relacionamento 1:N)
CREATE TABLE livro (
    id_livro SERIAL PRIMARY KEY,
    titulo VARCHAR(150),
    ano_publicacao INT,
    id_editora INT REFERENCES editora(id_editora)
);

-- Tabela Associativa Autor-Livro (Relacionamento N:N)
CREATE TABLE livro_autor (
    id_livro INT REFERENCES livro(id_livro),
    id_autor INT REFERENCES autor(id_autor),
    PRIMARY KEY (id_livro, id_autor) -- Chave Primária Composta
);

Nível 5: O Teste do Arquiteto (Boss)

Tempo estimado: 35 min

Leia a especificação abaixo. Mapeie as entidades, atributos e relacionamentos (no papel se preferir). Quando tiver o modelo lógico pronto, escreva o código SQL completo no terminal dinâmico abaixo para validar e visualizar a sua arquitetura de banco de dados real!

"Fomos contratados pela rede de Ginásios 'Iron-Fit'. O sistema deve registar os nossos Membros (precisamos do nome, cpf e telemóvel). Cada membro contrata obrigatoriamente um único Plano (ex: Mensal, Anual - guardamos o nome do plano e o valor), mas claro, um plano é assinado por vários membros.

O ginásio tem vários Instrutores (nome, especialidade). Um instrutor pode lecionar várias Aulas de Grupo (nome da aula, duração em minutos), mas cada aula específica só tem um instrutor responsável.

Por fim, os membros podem inscrever-se livremente nas aulas. Um membro faz várias aulas e uma aula tem vários membros."

solucao_boss.sql
-- Tabelas base (sem dependências)
CREATE TABLE plano (
    id_plano SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    valor DECIMAL(10,2)
);

CREATE TABLE instrutor (
    id_instrutor SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    especialidade VARCHAR(100)
);

-- Tabelas dependentes (Lado N dos relacionamentos 1:N)
CREATE TABLE membro (
    id_membro SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    cpf VARCHAR(14),
    telemovel VARCHAR(20),
    id_plano INT REFERENCES plano(id_plano)
);

CREATE TABLE aula (
    id_aula SERIAL PRIMARY KEY,
    nome VARCHAR(100),
    duracao_minutos INT,
    id_instrutor INT REFERENCES instrutor(id_instrutor)
);

-- Tabela Associativa (Relacionamento N:N entre membro e aula)
CREATE TABLE membro_aula (
    id_membro INT REFERENCES membro(id_membro),
    id_aula INT REFERENCES aula(id_aula),
    PRIMARY KEY (id_membro, id_aula)
);
