Aula 18: O Botão Vermelho

DDL Destrutivo: Aprenda a desmantelar estruturas, usar a opção nuclear com responsabilidade e limpar dados instantaneamente.

1. A Trindade da Destruição

Na arquitetura de software, às vezes é necessário demolir para poder reconstruir ou limpar o terreno. Muitos iniciantes confundem os comandos de exclusão, mas um Arquiteto sabe exatamente que ferramenta usar. Conheça a Trindade da Destruição:

DELETE DML

"O Bisturi"

  • Apaga dados (linhas) individualmente.
  • Permite usar WHERE (ex: apagar só o cliente X).
  • Lento em tabelas gigantes, pois regista cada linha apagada no log do sistema.
  • A tabela continua a existir.

TRUNCATE DDL

"O Aspirador Gigante"

  • Apaga TODOS os dados instantaneamente.
  • Não permite usar WHERE.
  • Super rápido, pois não regista linha a linha no log.
  • Reseta os contadores automáticos (SERIAL volta a 1).
  • A tabela (a estrutura) continua a existir.

DROP DDL

"A Bomba Nuclear"

  • Destrói A ESTRUTURA INTEIRA.
  • Os dados evaporam-se junto com a tabela.
  • As colunas, chaves e permissões deixam de existir.
  • Não sobra nada.

2. O Arsenal de Demolição

Existem diferentes níveis de destruição com o DROP. Conheça as variações utilizadas no mercado.

Básico

DROP TABLE

Destrói uma tabela isolada. Se houver alguma Chave Estrangeira (FK) de outra tabela a apontar para ela, o banco bloqueia a ação para proteger a integridade.

DROP TABLE logs_antigos;
Cascata

DROP ... CASCADE

A força bruta. Destrói a tabela forte e destrói automaticamente todas as pontes (Constraints/FKs) que outras tabelas tinham com ela.

DROP TABLE departamento CASCADE;
Apocalipse

DROP DATABASE

Apaga a base de dados inteira (todas as tabelas juntas). Atenção: Não pode executar este comando se estiver conectado dentro do próprio banco que quer apagar!

DROP DATABASE loja_db;

DROP TABLE IF EXISTS

No mundo real, os scripts SQL correm de forma automatizada em pipelines (DevOps). Se o script tentar executar um DROP TABLE numa tabela que já não existe, a base de dados devolve um erro fatal e paralisa a operação da empresa.

A cláusula IF EXISTS é a "trava de segurança". Se a tabela não estiver lá, o sistema ignora o comando educadamente em vez de quebrar.

A Dica de Ouro: O "Soft Drop"

Arquitetos seniores quase nunca dão um DROP TABLE direto numa base de dados de produção, mesmo que o CEO mande. Eles usam o Soft Drop: renomear a tabela para a "esconder".

ALTER TABLE clientes RENAME TO clientes_old_bkp;

Deixam a tabela lá por algumas semanas. Se nenhum sistema quebrar reclamando da falta dela, aí sim executam o DROP definitivo.

3. O Escudo Relacional (RESTRICT)

Lembra-se da Aula 17? Quando criamos uma Chave Estrangeira, o comportamento padrão do banco é o RESTRICT. Ele atua como um escudo. Se tentar usar um DROP TABLE simples numa tabela "Pai", o SGBD vai atirar-lhe um erro para impedir que as tabelas "Filhas" fiquem a apontar para o nada.

❌ A Forma Lenta (Segura)

Apagar primeiro as tabelas filhas, e só no fim apagar a tabela pai.

DROP TABLE funcionario;
DROP TABLE departamento;
🌪️ A Forma Rápida (CASCADE)

Diz ao banco: "Sei o que estou a fazer. Apaga a tabela e quebra as regras que a protegem".

DROP TABLE departamento CASCADE;
