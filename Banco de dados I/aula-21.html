Aula 21: O Sopro da Vida

DML e Inserção de Dados: A transição da fundação estrutural para o povoamento das tabelas utilizando o comando INSERT INTO.

1. A Casa está Pronta, mas está Vazia

Nas últimas 20 aulas, construímos o esqueleto do nosso sistema. Usámos comandos como CREATE TABLE e definimos as regras de aço (PKs, FKs, UNIQUE). Isso faz parte da Linguagem de Definição de Dados (DDL).

A partir de agora, o nosso papel muda. Deixamos de ser pedreiros e passamos a ser os gestores do conteúdo. Entramos no reino da Linguagem de Manipulação de Dados (DML), onde os dados reais (nomes, valores, datas) começam a fluir para dentro do banco de dados.

DDL (Definição)

Trabalha com a Estrutura.

CREATE, ALTER, DROP, TRUNCATE.

DML (Manipulação)

Trabalha com os Registos/Linhas.

INSERT, SELECT, UPDATE, DELETE.

2. A Sintaxe do INSERT INTO

O comando INSERT tem uma estrutura muito lógica: primeiro declaramos onde vamos inserir e em quais colunas, e de seguida os valores correspondentes na mesma ordem rigorosa.

A. Inserção Simples (Uma linha)

Atenção: Textos e Datas devem sempre estar entre aspas simples (' '). Números não precisam de aspas.

insert_simples.sql
INSERT INTO funcionario (nome, cargo, salario)
VALUES ('Ana Silva', 'Dev Frontend', 3500.50);
Curiosidade: Reparou que não inserimos o id_funcionario? Como a coluna foi criada como SERIAL (auto-incremento), o banco de dados gera o ID 1, 2, 3 automaticamente!

B. Inserção Múltipla (Bulk Insert)

Para inserir dezenas de registos, não repita o comando "INSERT INTO" várias vezes. Isso deixa o servidor lento. Basta separar os blocos de valores por vírgulas!

insert_multiplo.sql
INSERT INTO funcionario (nome, cargo, salario)
VALUES
  ('Rui Costa', 'Dev Backend', 4000.00),
  ('Maria João', 'UX Designer', 3200.00),
  ('Pedro Dias', 'DBA', 5500.00);

C. A Armadilha da Omissão (Perigo)

É possível omitir os nomes das colunas após o nome da tabela, mas o banco exigirá que passe todos os valores na ordem exata de criação da tabela. Em ambiente de produção, isto é uma bomba-relógio: se amanhã a sua equipa fizer um ALTER TABLE e adicionar ou reordenar uma coluna, todos os seus INSERTs vão quebrar instantaneamente!

nao_faca_isto.sql
-- MÁ PRÁTICA: Omitir a declaração das colunas.
INSERT INTO funcionario
VALUES ('João', 'Estagiário', 1000.00);

D. Migração em Massa (O Clone)

E se precisar de copiar 1 milhão de registos de uma tabela antiga para uma nova? Um Arquiteto de Dados não digita os VALUES à mão. Ele utiliza o poder do SELECT para injetar dados em massa de forma instantânea.

migracao.sql
INSERT INTO func_backup (nome, salario)
SELECT nome, salario FROM funcionario WHERE cargo = 'Dev Backend';

3. Regra de Ouro: A Ordem da Inserção

Quando o seu modelo de dados tem Chaves Estrangeiras (FK), a ordem de inserção é vital. Imagine tentar contratar um funcionário e colocá-lo no departamento de TI (ID: 5), mas você ainda não registou o departamento de TI na tabela de Departamentos!

O PostgreSQL rejeita a transação instantaneamente:
ERROR: insert or update on table "funcionario" violates foreign key constraint
A Solução: Preencha as tabelas de fora para dentro. Insira primeiro os registos nas tabelas "Pai" (ex: Departamento, Categoria) e só depois nas tabelas "Filhas" (ex: Funcionario, Produto).

O Teaser do Arquiteto: Tudo ou Nada (Transações)

Na vida real, se o sistema inserir o Departamento (Pai) com sucesso, mas a internet cair uma fração de segundo antes de inserir os Funcionários (Filhos), a base de dados ficará num estado inconsistente. Para evitar isso, os Arquitetos encapsulam múltiplos INSERTs num bloco de Transação (iniciando com BEGIN; e terminando com COMMIT;). Se algo falhar a meio, ocorre um ROLLBACK; e nada é guardado pela metade! Abordaremos isto a fundo no futuro.

4. O Elo com o Back-end (RETURNING)

Como aprendemos, quando utilizamos a coluna SERIAL, a base de dados (PostgreSQL/Neon) cria o ID sequencial automaticamente. No entanto, no cenário real, uma API em Node.js ou Python regista um novo Departamento e precisa desse ID recém-criado para associar-lhe novos funcionários logo de seguida. Como o Back-end descobre que ID foi gerado?

Para resolver este problema, utilizamos o superpoder exclusivo do PostgreSQL: a cláusula RETURNING. Ao adicioná-la no final do seu comando INSERT, a base de dados não só grava a linha, como a devolve imediatamente como resposta para a aplicação que fez o pedido.

api_backend.sql
INSERT INTO departamento (nome)
VALUES ('Inovação')
RETURNING id_depto; -- A BD devolve: { "id_depto": 6 } ao Node.js

5. O Escudo Anti-Colisão (ON CONFLICT)

Imagine que um utilizador na sua aplicação clica no botão "Registar" duas vezes muito rápido (o famoso duplo clique). O Back-end vai disparar dois comandos INSERT idênticos. O primeiro funciona, mas o segundo violará a regra de exclusividade (UNIQUE constraint) e devolverá um erro fatal, podendo causar a quebra (crash) da aplicação.

Para garantir a idempotência (o código pode correr 100 vezes sem destruir a base de dados), o PostgreSQL fornece a cláusula ON CONFLICT. É a base do conceito de Upsert.

idempotencia.sql
INSERT INTO funcionario (email, nome)
VALUES ('rui@mail.com', 'Rui Costa')
ON CONFLICT (email) DO NOTHING;

Tradução: "Tenta inserir o Rui. Se houver um conflito na coluna 'email' (porque ele já lá está), não dês erro nenhum, simplesmente ignora a instrução e segue a vida pacífica e silenciosamente."
