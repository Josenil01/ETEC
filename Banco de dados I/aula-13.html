Aula 13: A Arte da Purificação

Normalização Avançada e Desnormalização: Eleve as suas tabelas à 2ª e 3ª Formas Normais e saiba exatamente quando quebrar as regras.

Na Aula 12, curamos as nossas tabelas da 1FN: removemos listas, desmembramos moradas e garantimos que cada célula possuía apenas um valor atômico. Mas isso não é suficiente.

Mesmo na 1FN, os nossos dados ainda podem sofrer de redundância terrível (textos repetidos linha após linha). Para blindar o banco de dados completamente, precisamos de erradicar as Dependências Parciais (2FN) e as Dependências Transitivas (3FN).

1. A Segunda Forma Normal (2FN)

A Regra: Uma tabela está na 2FN se já estiver na 1FN e não contiver Dependências Funcionais Parciais.

Em português simples: Se a sua tabela tem uma Chave Primária Composta (feita de 2 ou mais colunas), todos os outros atributos da tabela têm de depender da chave INTEIRA, e não apenas de um pedaço dela.

Regra de Ouro (Atalho Mental)

Se uma tabela já está na 1FN e possui uma Chave Primária Simples (apenas uma coluna), ela está automaticamente na 2FN! Só precisa se preocupar com a 2FN se houver chaves compostas.

2. A Terceira Forma Normal (3FN)

A Regra: Uma tabela está na 3FN se estiver na 2FN e não contiver Dependências Funcionais Transitivas.

"Tudo depende da chave, de toda a chave, e de nada mais além da chave. Que Codd me ajude."

Em português simples: Um atributo que não é chave primária NÃO PODE depender de outro atributo que também não é chave primária. Se a coluna A depende da coluna B, que depende da PK, temos um intruso na tabela!

O Grande Tira-Teima: 2FN vs. 3FN

Problema da 2FN

Envolve a Chave Primária. Acontece quando a chave é composta (duas ou mais colunas) e o dado depende de apenas METADE dessa chave.

Problema da 3FN

Envolve Atributos Comuns. Acontece quando um dado "comum" manda noutro dado "comum", ignorando a chave primária completamente.

3. Desnormalização: Quebrando as Regras

Normalizar até à 3FN deixa o banco de dados livre de anomalias, mas cria dezenas de tabelas. Na hora de consultar um Pedido, o banco precisará fazer múltiplos JOINs para reunir o Nome do Cliente, o Nome do Produto, o Preço, etc. Se a loja tiver milhões de acessos, o desempenho pode cair drasticamente.

Quando Desnormalizar (Duplicar dados de propósito)?

  • Para Performance (Relatórios): Guardar o `Total_Venda` na tabela de Vendas, em vez de o calcular sempre somando os itens.
  • Para Precisão Histórica: Copiar o `Preco_Atual` do Produto para dentro da tabela `TB_ITENS_VENDA` no momento da compra. (Isto fere a 3FN, mas garante que se o preço subir amanhã, o recibo de ontem não seja alterado!).

Os Perigos Sombrios (O Preço a Pagar)

A desnormalização não é de graça. Ao duplicar dados para ganhar performance, você traz de volta o fantasma das Anomalias de Atualização.

  • Se você criar uma coluna Total_Venda, o sistema precisará de lógicas super rígidas no Back-end ou de Triggers (Gatilhos) no banco para recalcular esse total sempre que um item for adicionado ou removido.
  • Se a regra falhar, o seu banco de dados exibirá informações financeiras mentirosas!

5. Mão na Massa (Prática SQL)

Evolua o nosso sistema de Vendas para a 3FN. Crie 4 tabelas interligadas para resolver todos os problemas de redundância: cliente, produto, pedido e item_pedido.

Dica de Ouro (Desnormalização): Lembre-se de colocar a coluna preco_vendido na tabela item_pedido, quebrando as regras para salvar o histórico!

solucao_3fn.sql
CREATE TABLE cliente (
    id_cliente SERIAL PRIMARY KEY,
    nome VARCHAR(100)
);

CREATE TABLE produto (
    id_produto SERIAL PRIMARY KEY,
    nome_produto VARCHAR(100),
    preco_atual DECIMAL(10,2)
);

CREATE TABLE pedido (
    id_pedido SERIAL PRIMARY KEY,
    id_cliente INT REFERENCES cliente(id_cliente),
    data_pedido DATE
);

CREATE TABLE item_pedido (
    id_pedido INT REFERENCES pedido(id_pedido),
    id_produto INT REFERENCES produto(id_produto),
    quantidade INT,
    preco_vendido DECIMAL(10,2), -- Desnormalização histórica
    PRIMARY KEY (id_pedido, id_produto)
);
