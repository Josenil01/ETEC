Otimização com Índices: Aprenda a acelerar consultas em grandes volumes de dados.

1. Como o Índice funciona?

Um índice funciona como o índice remissivo no final de um livro de 1000 páginas. Sem ele, para encontrar um assunto, o banco de dados precisa ler a tabela linha a linha do início ao fim (fenómeno chamado de Sequential Scan). Com ele, o banco salta diretamente para a página exata usando uma estrutura inteligente (como uma Árvore-B), realizando um Index Scan.

O Trade-off do Arquiteto: Índices aceleram drasticamente a leitura (SELECT), mas tornam a escrita (INSERT/UPDATE/DELETE) mais lenta, pois a árvore do índice precisa ser recalculada a cada nova alteração. Use-os com sabedoria!

Curiosidade do Arquiteto (Além da Árvore-B): O PostgreSQL possui outros motores mágicos! Sabia que pode usar o índice GIN para procuras ultrarrápidas dentro de textos longos ou documentos JSON, e o GiST para procurar dados geográficos (como coordenadas de mapas)? A Árvore-B é o padrão, mas o Arquiteto escolhe a ferramenta certa para o problema.

2. O Raio-X do Arquiteto (EXPLAIN)

Como é que um Arquiteto Sênior prova que a sua consulta está rápida? Ele não usa o cronômetro do telemóvel; ele usa o comando EXPLAIN ANALYZE antes do `SELECT`. O PostgreSQL executa a query e devolve o seu "plano de voo", revelando exatamente como encontrou os dados.

EXPLAIN ANALYZE SELECT * FROM funcionario WHERE email = 'clara@empresa.com';

-- 🔴 DIAGNÓSTICO SEM ÍNDICE (Desespero):
Seq Scan on funcionario (cost=0.00..15233.00 rows=1 width=128)
  Filter: ((email)::text = 'clara@empresa.com'::text)
Execution Time: 998.50 ms

-- 🟢 DIAGNÓSTICO COM ÍNDICE (Paz de Espírito):
Index Scan using idx_funcionario_email on funcionario (cost=0.42..8.44 rows=1)
  Index Cond: ((email)::text = 'clara@empresa.com'::text)
Execution Time: 0.04 ms

Como ler este relatório?

O Método (Seq Scan vs Index Scan)

É a parte mais importante. Seq Scan significa "Busca Sequencial" (leu a tabela toda). Index Scan significa que o banco encontrou o atalho na Árvore-B.

Cost (ex: 0.42..8.44)

Não são segundos! É uma pontuação interna de "esforço" do processador. O 1º número é o custo para achar a 1ª linha; o 2º é o custo total. Quanto menor, melhor.

Rows (ex: rows=1)

Mostra quantas linhas o banco de dados estimou que teria de verificar para concluir a pesquisa.

Execution Time

É a sua Métrica de Ouro. Representa o tempo real que a consulta demorou em milissegundos (ms).

Na Prática: Para usar isto no SQL Editor do Neon, basta escrever EXPLAIN ANALYZE logo antes de qualquer SELECT, UPDATE ou DELETE e executar!

A Armadilha Invisível (Funções cegam Índices)

Imagine que você criou o índice na coluna email. Mas na aplicação (Node.js/Python), o código pesquisa assim: WHERE UPPER(email) = 'CLARA@EMPRESA.COM'.

O que acontece? O banco de dados faz um Seq Scan de 1 segundo! Quando você aplica uma função como UPPER() na coluna, o banco não consegue usar a árvore original. A cura para isto é não usar funções na coluna durante o `WHERE`, ou criar um Índice Baseado em Função.

Dica: Ctrl+Alt+X

Conheça o kit de ferramentas DDL completo do Arquiteto para manipulação de Índices:

Clássico
Índice Simples CREATE INDEX idx_email

Combo
Índice Composto CREATE INDEX idx_nome_sobrenome

Limpeza
Remover Índice DROP INDEX idx_email;
Económico
Índice Parcial CREATE INDEX idx_atv
ON func(email)
WHERE ativo = true;

Poupa muito espaço em disco ignorando as linhas irrelevantes.

Desafio 2: A Query Misteriosa

Análise de Gargalo

Um programador júnior está em pânico. Ele criou um índice certinho na coluna cpf para acelerar as buscas no sistema. No entanto, o `EXPLAIN ANALYZE` provou que a query abaixo está demorando 5 segundos (fazendo Seq Scan) numa tabela de 10 milhões de registros!

SELECT * FROM cliente WHERE REPLACE(cpf, '.', '') = '12345678900';

Baseado no que aprendeu hoje, por que o índice foi sumariamente ignorado e qual seria a solução?

O Erro (A Armadilha Invisível): Ao utilizar a função REPLACE() diretamente na coluna cpf dentro da cláusula WHERE, o programador cegou a base de dados. O índice conhece apenas os CPFs com os pontos originais. Para verificar essa condição, o banco de dados é forçado a ler todas as linhas da tabela (Seq Scan) para aplicar a função e só depois comparar.

As Soluções:

  • Solução 1 (Melhor): Limpar a formatação do CPF no próprio código (Node.js/Python) antes de enviar a consulta SQL ao banco.
  • Solução 2 (Avançada): Criar um Índice Baseado em Função para guardar o valor já limpo na árvore B-Tree:
    CREATE INDEX idx_cpf_limpo ON cliente(REPLACE(cpf, '.', ''));

Desafio 3: O Pesadelo do Relatório

Tuning Avançado

Todo final de mês, o sistema de vendas trava. O diretor financeiro executa um relatório pesado buscando as vendas concluídas em um período específico:

SELECT * FROM venda
WHERE status = 'CONCLUIDO'
AND data_venda BETWEEN '2026-01-01' AND '2026-01-31';

O antigo DBA criou um índice apenas para a coluna status (idx_status_velho), mas como 90% das vendas são "CONCLUIDO", esse índice não ajuda a filtrar quase nada.

Sua missão: Destrua o índice antigo inútil e crie um Índice Composto poderoso que cubra simultaneamente as colunas status e data_venda da tabela venda.

Atenção / Regra de Ouro (Left-to-Right)

A ordem das colunas num índice composto é vital! Um índice criado como (status, data_venda) acelera procuras efetuadas por status ou por status + data. No entanto, é totalmente inútil se a procura for apenas por data_venda. É tal e qual como procurar o apelido de alguém numa lista telefónica que está organizada pelo primeiro nome!

tuning_vendas.sql 
DROP INDEX idx_status_velho;

CREATE INDEX idx_venda_status_data 
ON venda(status, data_venda);
