Módulo 5: Manipulação de Dados (DML) Arquiteto de Base de Dados

Aula 23: O Ceifador

DML e Exclusão de Dados: O comando DELETE e como a Integridade Referencial o impede de destruir o ecossistema acidentalmente.

1. A Sintaxe da Exclusão

Já demos vida aos dados (INSERT) e já os curámos/alterámos (UPDATE). Mas, inevitavelmente, registos tornam-se obsoletos, clientes cancelam contas ou produtos saem de linha. O comando da linguagem de manipulação (DML) para apagar registos específicos é o DELETE FROM.

Estrutura do Comando
DELETE FROM funcionario

1. Qual é a tabela que vai perder dados?

WHERE id_func = 4;

2. O Filtro: QUEM será apagado? (Use SEMPRE a Chave Primária!)

O Irmão do UPDATE sem WHERE

Tal como no UPDATE, se executar um DELETE FROM funcionario; SEM A CLÁUSULA WHERE, não apaga um funcionário. Apaga TODOS os funcionários da empresa num milissegundo. Cuidado extremo!

Nível Sénior

O Segredo da Indústria: Soft Delete (Exclusão Lógica)

No mundo real (Bancos, Hospitais, E-commerce), nós quase nunca usamos o DELETE físico. Se apagar definitivamente um cliente da base de dados, todos os recibos, faturas e históricos de compras ligados a ele ficarão corrompidos ou órfãos. Além disso, leis de proteção de dados e auditorias exigem a preservação de rastos.

-- Em vez de usar DELETE, os Arquitetos adicionam uma coluna 'ativo' (BOOLEAN) e fazem um UPDATE:
UPDATE cliente SET ativo = FALSE WHERE id_cliente = 4;

Isto chama-se Soft Delete (Exclusão Lógica). O registo "desaparece" para o utilizador final na aplicação, mas continua guardado em segurança no servidor para efeitos de auditoria.

2. O Escudo (Integridade Referencial)

Você tenta apagar o Departamento de TI (ID = 1). Mas o Rui e a Ana trabalham lá! O que acontece no mundo real?

A Muralha (RESTRICT)

A base de dados padrão tem a restrição ON DELETE RESTRICT ativa pelas Chaves Estrangeiras (FK). Ela age como um escudo de força: Bloqueia a exclusão para proteger os filhos (funcionários).

consola.sql
DELETE FROM departamento WHERE id_depto = 1;

ERROR: update or delete on table "departamento" violates foreign key constraint
DETAIL: Key (id_depto)=(1) is still referenced from table "funcionario".

Como o Arquiteto Resolve?

Para apagar o Departamento 1 de forma limpa quando o sistema bloqueia, tem de agir nas tabelas "Filhas" primeiro. Existem duas soluções táticas:

  • Demitir todos (DELETE): Faz um DELETE FROM funcionario WHERE id_depto = 1;. Agora o departamento está vazio e pode ser apagado.
  • Transferir todos (UPDATE): Faz um UPDATE funcionario SET id_depto = 2 WHERE id_depto = 1;. O departamento 1 ficou vazio, pode apagá-lo em paz.

O Efeito Dominó (ON DELETE CASCADE)

Se, na Aula 17, tivéssemos configurado a FK com ON DELETE CASCADE, a base de dados não daria erro. Ela apagaria o Departamento 1 e mataria instantaneamente o Rui e a Ana do sistema de forma automática. Nunca use CASCADE para pessoas, apenas para coisas descartáveis (ex: Apagar um Post apaga os seus Comentários).

3. O Último Suspiro (RETURNING)

Tal como vimos no INSERT e no UPDATE, a cláusula RETURNING também é um superpoder no DELETE. Se a sua API em Back-end apagar um funcionário, ela muitas vezes precisa de mostrar um aviso de sucesso com o nome da pessoa eliminada.

Como o Back-end sabe o nome se o registo acabou de ser apagado? Simples: a base de dados "cospe" os dados de volta numa fração de segundo antes de eles desaparecerem para sempre no abismo.

api_backend.sql
DELETE FROM funcionario WHERE id_func = 2
RETURNING nome, id_depto; -- O Node.js recebe: { "nome": "Rui Costa", "id_depto": 1 }

4. Exclusão Relacional (A Cláusula USING)

Nível Mestre

E se o chefe pedir para apagar todos os funcionários do departamento de "TI", mas você não sabe qual é o ID do departamento? Num SELECT, usaríamos um JOIN, mas o DELETE padrão não suporta o comando JOIN diretamente.

A solução avançada exclusiva do PostgreSQL é a cláusula USING. Ela permite referenciar tabelas adicionais no momento de apagar, fazendo um "cruzamento de dados" destrutivo.

delete_using.sql
DELETE FROM funcionario
USING departamento -- A tabela de referência
WHERE funcionario.id_depto = departamento.id_depto -- O cruzamento
AND departamento.nome = 'TI';

5. O Arquivamento Perfeito (A Magia das CTEs)

Embora o Soft Delete seja a regra, às vezes precisamos realmente de limpar a tabela principal para otimizar o desempenho, movendo as linhas antigas para uma tabela de Histórico/Arquivo.

Fazer um SELECT (para ler), depois um INSERT (para copiar) e por fim um DELETE (para apagar) é perigoso: o sistema pode falhar a meio. O padrão absoluto da indústria é combinar o RETURNING com as Expressões de Tabela Comuns (CTEs), movendo os dados atómicamente numa única viagem!

arquivo_atomico.sql
WITH apagados AS ( -- Cria uma "memória temporária"
    DELETE FROM funcionario WHERE id_func = 5
    RETURNING * -- Salva a linha apagada na memória
)
INSERT INTO historico_funcionario
SELECT * FROM apagados; -- Injera os dados salvos no arquivo

6. Laboratório de Exclusão

Dica: Ctrl+Alt+X

À direita, tem a visão real do nosso banco de dados relacional. Assuma a consola e efetue as remoções solicitadas.

1

O Corte Cirúrgico

O funcionário Leo (ID 4) pediu demissão. Como ele é um registo "Filho" (não tem tabelas abaixo dele), a exclusão será pacífica. Apague-o usando a sua PK.

DELETE FROM funcionario WHERE id_func = 4;
2

A Muralha de Integridade

A direção mandou fechar o departamento de TI (ID 1). Tente executar o comando DELETE para apagar o departamento e veja o que a base de dados (o escudo RESTRICT) lhe diz.

DELETE FROM departamento WHERE id_depto = 1;
3

Exclusão Segura

O escudo parou-o no passo anterior. Mas repare no departamento Inovação (ID 6). Ele não tem nenhum funcionário vinculado (órfão invertido). Apague o ID 6 e veja a operação funcionar com sucesso.

DELETE FROM departamento WHERE id_depto = 6;
4

O Último Suspiro

Apague o funcionário Max (ID 9). No mesmo comando, utilize a cláusula mágica estudada para capturar o seu nome e a sua chave estrangeira, escrevendo exatamente: RETURNING nome, fk_depto; para a API capturar os dados na queda.

DELETE FROM funcionario WHERE id_func = 9 RETURNING nome, fk_depto;
5
Ultimate Boss

O Arquivamento Perfeito

Mova a funcionária Zoe (ID 8) para a tabela imaginária tb_arquivo e apague-a da tabela principal na mesma transação. Use o poder da CTE (WITH apagados AS (...)).

WITH apagados AS ( DELETE FROM funcionario WHERE id_func = 8 RETURNING * ) INSERT INTO tb_arquivo SELECT * FROM apagados;
