Módulo 3: Infraestrutura e Nuvem Arquiteto de Banco de Dados

Aula 17: O Efeito Dominó

DDL Avançado: Como conectar tabelas existentes, garantir a integridade referencial e dominar a reação em cadeia da exclusão de dados.

1. O Dilema das Ilhas Isoladas

Na Aula 16, construímos a nossa base no Neon criando as tabelas Departamento e Funcionario separadamente. Neste momento, o banco de dados vê essas duas tabelas como ilhas isoladas.

Qual é o perigo disso?

Sem a ponte da Chave Estrangeira (FK), o sistema permitiria que você contratasse um funcionário e o colocasse no departamento de código 999 (um departamento que nem sequer existe!). O banco não faria nenhuma validação, corrompendo as regras do negócio.

2. Integridade e o Efeito Dominó

Antes de construirmos a ponte, precisamos de decidir a sua física. O que acontece com os Funcionários se a empresa decidir apagar o Departamento de RH? Existem 3 comportamentos (ações) possíveis que você, como Arquiteto, deve escolher:

Padrão

RESTRICT

"O Escudo de Proteção"

Se tentar apagar o RH e existirem funcionários lá dentro, o banco aborta a operação e grita um erro. Para apagar o RH, teria primeiro de transferir ou demitir os funcionários.

Perigoso

CASCADE

"A Reação em Cadeia"

Se apagar o RH, o banco vai apagar automaticamente todos os funcionários vinculados a esse departamento. Ideal para relação Pedido -> Itens_do_Pedido (se cancelar o pedido, apaga os itens).

Flexível

SET NULL

"O Orfanato"

O RH é apagado. O banco vai aos funcionários do RH e altera a coluna id_depto deles para NULL (vazio). Eles ficam órfãos, aguardando realocação.

3. A Arte do ALTER TABLE

O comando ALTER TABLE é o "canivete suíço" do Arquiteto de Dados. No mundo real, as regras de negócio mudam constantemente: os clientes pedem novos campos, removem antigos ou precisam de conectar tabelas que antes estavam isoladas. Ele serve tanto para evoluir a estrutura como para criar pontes.

A Evolução da Estrutura (Colunas)

Adicionar Campo

O cliente pediu para adicionar a "data de nascimento" aos funcionários.

ALTER TABLE funcionario
ADD COLUMN data_nasc DATE;

Remover Campo

O campo "fax" tornou-se obsoleto e precisa de ser eliminado permanentemente.

ALTER TABLE funcionario
DROP COLUMN fax;

Renomear Campo

Padronização: mudar o nome do campo "morada" para "endereco".

ALTER TABLE funcionario
RENAME COLUMN morada TO endereco;

Alterar Tipo de Dado

O "salario" que era um número simples (INT) agora precisa de decimais (DECIMAL).

ALTER TABLE funcionario
ALTER COLUMN salario TYPE DECIMAL(10,2);

A Criação de Pontes (Constraints)

Para conectarmos as tabelas já existentes, usamos a instrução de alteração. Note que, como boas práticas de arquitetura, nós damos um nome à nossa ponte (Constraint) para podermos geri-la ou apagá-la no futuro.

ALTER TABLE funcionario
ADD CONSTRAINT fk_func_departamento -- Nome da ponte
FOREIGN KEY (id_depto) -- Coluna na tabela funcionário
REFERENCES departamento(id_depto) -- Onde ela deve procurar o alvo
ON DELETE SET NULL; -- A física escolhida

Perigo: ADD COLUMN + NOT NULL

Se adicionar uma nova coluna com a restrição NOT NULL a uma tabela que já possui milhares de registos, o banco abortará a operação! Porquê? Porque os registos antigos ficariam vazios (NULL) nessa nova coluna.
Solução: Adicione a coluna permitindo nulos, atualize os dados antigos para valores padrão e só depois aplique o NOT NULL.

Destruindo Pontes (DROP CONSTRAINT)

Criou uma Chave Estrangeira errada? Não precisa de destruir a tabela inteira. Como seguimos as boas práticas e demos um nome à nossa ponte (ex: fk_func_departamento), basta usar este comando para a eliminar:
ALTER TABLE funcionario DROP CONSTRAINT fk_func_departamento;
