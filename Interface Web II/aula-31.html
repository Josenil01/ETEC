1. A Armadilha do Charada (SQL Injection)

O Charada enviou uma mensagem criptografada para o Batcomputador. Para abri-la, o Batman precisa inserir a senha correta. Mas há um problema crítico na forma como sistemas iniciantes consultam o banco de dados.

Muitos sistemas apenas "colam" (concatenam) o que o usuário digita diretamente no comando SQL. Se o Charada digitar algo como ' OR '1'='1, o banco de dados se confunde. Ele não sabe onde termina o "dado" e onde começa o "comando".

// O que o programador queria fazer:
SELECT * FROM arquivos WHERE senha = 'senha_aqui';

// O que acontece quando o Charada ataca:
SELECT * FROM arquivos WHERE senha = '' OR '1'='1';

Como 1=1 é sempre verdadeiro na matemática, o banco de dados ignora a senha e devolve todos os arquivos secretos de Arkham. O sistema foi invadido. A isto chamamos de SQL Injection.

2. A Porta de Entrada (Como o ataque viaja?)

O Front-end (HTML/JavaScript) é apenas a "caixa de correio". Ele não processa o banco de dados e não sofre o ataque diretamente. O Charada usa elementos comuns da interface web para despachar sua carga maliciosa até o servidor (Back-end). Os caminhos mais usados são:

  • Campos de Formulário (Inputs): O invasor digita o código SQL em telas de login, barras de pesquisa ou formulários de cadastro.
  • Parâmetros na URL: O invasor altera endereços legítimos (ex: site.com/perfil?id=5) adicionando código extra (ex: site.com/perfil?id=5 OR 1=1).

Simulador do Batcomputador: O Ataque vs A Defesa

❌ Sistema Vulnerável (Concatenação) SELECT * FROM arquivos WHERE senha = '' OR '1'='1'

Resultado: O banco entende isso como um COMANDO e libera o acesso total!

✅ Sistema Seguro (Parametrização) EXECUTE busca_segura(' OR '1'='1)

Resultado: O banco trata a entrada apenas como um TEXTO puro. Acesso Negado!

3. O Cinto de Utilidades (A Solução)

A defesa definitiva contra esse ataque chama-se Consultas Parametrizadas (ou Prepared Statements). A ideia é criar um "molde" fixo no banco de dados primeiro. O banco analisa o molde e trava a estrutura. Só depois enviamos o dado do utilizador para preencher a lacuna (o parâmetro $1).

Neon.tech 1. A Preparação da Defesa (SQL Nativo)

Abra o editor SQL do Neon.tech. Primeiro, vamos criar a tabela. Em seguida, usamos o comando PREPARE. O $1 avisa o banco de dados: "Atenção, aqui entrará APENAS texto simples, nunca um comando".

console.sql
-- 1. Criando a base de dados secreta
CREATE TABLE arquivos_secretos (
    id SERIAL PRIMARY KEY,
    senha_acesso VARCHAR(100),
    mensagem_secreta TEXT
);

INSERT INTO arquivos_secretos (senha_acesso, mensagem_secreta) 
VALUES ('morcego123', 'O ataque será no banco de Gotham.');

-- 2. O Batman cria o "Molde Impenetrável" (Prepared Statement)
PREPARE busca_segura (text) AS
SELECT mensagem_secreta FROM arquivos_secretos WHERE senha_acesso = $1;

Neon.tech 2. Testando o Ataque no Banco

Com o molde pronto, usamos o EXECUTE para inserir os dados. Veja como o banco reage à senha correta e ao ataque.

console.sql
-- Teste 1: O Batman inserindo a senha correta
-- O sistema substitui o $1 por 'morcego123' e revela a mensagem.
EXECUTE busca_segura('morcego123');

-- Teste 2: O ataque do Charada neutralizado!
-- O Charada envia seu código malicioso. Mas graças ao $1, o banco 
-- procura literalmente por uma senha que se chame "' OR '1'='1".
-- Como essa senha absurda não existe, ele bloqueia o acesso!
EXECUTE busca_segura(''' OR ''1''=''1');

4. Laboratório Real: Criando e Corrigindo a Falha

Agora vamos colocar a mão na massa construindo um servidor Node.js real, conectado ao seu Neon.tech. Você vai programar a vulnerabilidade de propósito, hackear seu próprio sistema e, em seguida, aplicar a vacina.

1 Prepare o Terreno no Neon.tech

Abra o SQL Editor do seu projeto Neon e rode o comando abaixo para criar a tabela de usuários com um administrador secreto:

CREATE TABLE usuarios_arkham (
    id SERIAL PRIMARY KEY,
    usuario VARCHAR(50),
    senha VARCHAR(50)
);
INSERT INTO usuarios_arkham (usuario, senha) VALUES ('batman', 'morcego123');

2 O Servidor Back-end (Node.js)

No seu terminal local (VS Code), crie uma pasta, inicialize o projeto e instale as bibliotecas necessárias:

> npm init -y
> npm install express pg

Crie o arquivo server.js e cole o código abaixo. Preste atenção nas duas rotas que criamos: uma com a falha e outra protegida.

server.js
const express = require('express');
const { Pool } = require('pg');
const app = express();

// Configuração crítica para que o Express consiga ler os dados do formulário POST
app.use(express.urlencoded({ extended: true }));

// Substitua pela sua string de conexão do Neon.tech
const pool = new Pool({
    connectionString: 'postgres://seu_usuario:sua_senha@ep-seu-banco.neon.tech/neondb?sslmode=require'
});

// 🔴 ROTA 1: A VULNERABILIDADE (O Erro do Estagiário)
app.post('/login-falho', async (req, res) => {
    const user = req.body.user;
    const pass = req.body.pass;
    
    // PERIGO: Concatenando diretamente o que veio do formulário!
    const queryVulneravel = `SELECT * FROM usuarios_arkham WHERE usuario = '${user}' AND senha = '${pass}'`;
    
    try {
        const resultado = await pool.query(queryVulneravel);
        if(resultado.rows.length > 0) res.send("<h1 style='color:red'>ACESSO TOTAL CONCEDIDO: </h1>" + JSON.stringify(resultado.rows));
        else res.send("Acesso Negado.");
    } catch(err) { res.send("Erro no banco: " + err.message); }
});

// 🟢 ROTA 2: A CORREÇÃO (Consultas Parametrizadas)
app.post('/login-seguro', async (req, res) => {
    const user = req.body.user;
    const pass = req.body.pass;
    
    // DEFESA: Usamos $1 e $2 como moldes vazios. O banco de dados sabe que isso é só texto.
    const querySegura = `SELECT * FROM usuarios_arkham WHERE usuario = $1 AND senha = $2`;
    
    try {
        // Passamos os dados separadamente dentro de um Array [user, pass]
        const resultado = await pool.query(querySegura, [user, pass]);
        if(resultado.rows.length > 0) res.send("<h1 style='color:green'>ACESSO CONCEDIDO (SEGURO): </h1>" + JSON.stringify(resultado.rows));
        else res.send("Acesso Negado. Tentativa bloqueada!");
    } catch(err) { res.send("Erro no banco: " + err.message); }
});

app.listen(3000, () => console.log('Batcomputador rodando em http://localhost:3000'));

Rode o servidor executando: node server.js

Dica de Segurança: GET vs POST

Reparou que mudámos as nossas rotas de app.get() para app.post()? Nunca envie palavras-passe via GET! No método GET, os dados são enviados na própria URL do navegador (ex: ?pass=minhasenha), ficando expostos no histórico e nos ecrãs de quem passar perto. O POST envia os dados no "corpo" da requisição, de forma invisível na barra de endereços.

💣 3. Explore a Vulnerabilidade

Com o servidor a correr, abra o seu ficheiro index.html (criado na Secção 3) no navegador. Deixe o utilizador como "batman" e digite o ataque no campo da chave:

' OR '1'='1

O que acontece? Ao clicar em Autenticar, o formulário envia os dados via POST. O JavaScript cola o código do Charada, montando a query senha = '' OR '1'='1' e destrói a segurança do sistema. Receberá o ecrã vermelho de acesso concedido!

🛡️ 4. Teste a Correção

No seu index.html, altere a tag form para apontar para a rota segura: action="http://localhost:3000/login-seguro". Recarregue a página e tente o mesmo ataque:

' OR '1'='1

O que acontece? Receberá "Acesso Negado". A biblioteca pg garantiu que a frase do Charada fosse interpretada literalmente como uma senha absurda graças aos parâmetros $1 e $2.
