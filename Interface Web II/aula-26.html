Aula 26: Os Arquivos Jedi

O conhecimento solto perde-se. Aprenda a usar import, export e a separar o seu código em Módulos (Camada Visual vs Camada de Negócio).

1. O Código Esparguete e o Lado Negro

Na Aula 25, escrevemos a Classe Padawan, manipulámos botões do HTML (DOM), criámos eventos e atualizámos estilos, tudo dentro do mesmo ficheiro app.js. Para um projeto pequeno, isso funciona. Mas à medida que a aplicação cresce, ter 5.000 linhas de código no mesmo sítio torna-se num caos ilegível conhecido como "Código Esparguete" (Spaghetti Code).

A solução da Ordem Jedi é a Modularização. Nos Arquivos de Coruscant, a informação é separada em diferentes Holocrons (Ficheiros). Hoje vamos aprender a criar um ficheiro só para as Classes (A Regra de Negócio) e outro ficheiro só para mexer no HTML (A Interface/UI), ligando-os através de Módulos ES6.

2. A Sintaxe dos Holocrons

O Emissor (export)

Por padrão, tudo o que escreve num ficheiro JS está "trancado" lá dentro. A palavra export avisa o JavaScript: "Autorizo que outros ficheiros usem esta Classe/Variável".

export class Padawan {
  // Lógica aqui
}
O Recetor (import)

No seu ficheiro principal (onde vai manipular o ecrã), você usa a palavra import para ir buscar as peças aos outros ficheiros, indicando o caminho exato.

import { Padawan } from './Jedi.js';

const luke = new Padawan();
A Chave Mestra (HTML)

Se tentar usar import num script normal, o navegador vai explodir com um erro vermelho. É obrigatório avisar o HTML que esse script agora é um "Módulo".

<script type="module"
src="main.js"></script>

3. Terminal dos Arquivos (Preview)

Neste simulador holográfico, o clique num botão no módulo "Interface (main.js)" aciona a lógica que está encapsulada no módulo "Regras (Jedi.js)". A separação é invisível para o utilizador, mas essencial para o Arquiteto.

A Arquitetura de Separação

Aviso de Segurança (CORS) e A Árvore do Projeto

Devido às rigorosas políticas de segurança dos navegadores, ficheiros que utilizam a diretiva type="module" não podem ser abertos com um simples duplo-clique no computador. É obrigatório utilizar um servidor local. Garanta que está a usar o Live Server!

A Estrutura Lógica: Todos os ficheiros devem estar juntos na mesma pasta! ├── 📄 index.html   (A Interface Visual)
├── 📄 Jedi.js      (A Regra de Negócio / A Classe Oculta)
└── 📄 main.js      (O Controlador que os une usando imports)

HTML 1. A Chave de Permissão

No seu index.html, a única coisa que muda em relação às aulas normais é a tag do script. O atributo type="module" é obrigatório. Sem ele, o navegador proíbe o uso de imports. Adicionamos também uma <div> vazia para mostrar os resultados.

index.html
<body>
    <div id="ui-app">
        <button id="meu-botao">Criar Jedi</button>
        
        <!-- Div para injetar o resultado no ecrã (Fim dos Alerts!) -->
        <div id="resultado-jedi"></div>
    </div>

    <!-- ATENÇÃO AO TYPE="MODULE" -->
    <script type="module" src="main.js"></script>
</body>

Camada de Negócio 2. Ficheiro Jedi.js (O Molde)

Este ficheiro é 100% puro JavaScript. Não tem nenhuma ligação com o HTML (não há getElementById, não há querySelector). A sua única função é definir as regras lógicas e métodos práticos, e exportá-los.

Jedi.js
// Usamos 'export' antes da classe para torná-la pública
export class Padawan {
    #nivelForca;

    constructor(nome) {
        this.nome = nome;
        this.#nivelForca = 10;
    }

    consultarForca() {
        return this.#nivelForca;
    }

    // Adicionamos um método utilitário prático
    ativarSabre() {
        return "Bzzzz! Sabre de luz ativado!";
    }

    treinar(pontos) {
        if (typeof pontos !== 'number') return;
        this.#nivelForca += pontos;
    }
}

Camada Visual 3. Ficheiro main.js (O Controlador)

Este ficheiro foca-se em interagir com o DOM (Interface). Ele importa a lógica de negócio do Jedi.js, instancia o Objeto, usa os seus métodos, e injeta os resultados no HTML sem usar alertas intrusivos!

main.js
// 1. Trazemos a Classe de fora (Atenção ao ./ e ao .js no final)
import { Padawan } from './Jedi.js';

// 2. Lógica Visual (DOM)
const botao = document.getElementById('meu-botao');
const resultado = document.getElementById('resultado-jedi');

botao.addEventListener('click', () => {
    // 3. Usamos o Molde que importámos!
    const meuJedi = new Padawan('Ahsoka Tano');
    
    meuJedi.treinar(50);
    
    // Feedback na Tela usando manipulação de DOM (Adeus alert!)
    resultado.innerHTML = `
        <br>A Jedi <strong>${meuJedi.nome}</strong> foi criada via Módulo!<br>
        Ação: ${meuJedi.ativarSabre()}<br>
        Força: ${meuJedi.consultarForca()}
    `;
    resultado.style.color = "#06b6d4"; // Ciano brilhante
});

4. Provações de Coruscant (Prática)

01
A Forja do Sabre (Export):

Crie um novo ficheiro chamado Sabre.js. Lá dentro, crie uma simples função que retorna um texto: function ligarSabre() { return "Vrummm!"; }. Adicione a palavra-chave correta antes de function para que ela possa ser acedida por outros ficheiros.

02
Empunhando o Sabre (Import):

No seu ficheiro main.js, vá para a linha 1 e importe a função ligarSabre do ficheiro Sabre.js. A seguir, chame a função dentro de um console.log() para ouvir o barulho do sabre na sua consola.

03
O Export Default (Desafio Avançado)

Até agora usámos Named Exports (com chaves {}). Mas um ficheiro pode ter um ator principal. No seu Jedi.js, altere o código para: export default class Padawan....

O que muda no main.js? Agora tem de importar sem as chaves! Modifique o seu import para import Padawan from './Jedi.js';. Teste no navegador. Esta é a sintaxe mais usada em frameworks modernos como o React!
