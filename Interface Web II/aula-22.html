Aula 22: A Bigorna

Melhore e modifique os seus itens. Aprenda a operação de Atualização (Update) usando a cláusula WHERE para não destruir o mundo.

1. Reforjando Dados (CRUD)

Já aprendemos a criar (C) e a ler (R). Mas o que acontece quando um jogador melhora uma "Espada de Madeira" para uma "Espada de Ferro"? Não precisamos de apagar a espada antiga e criar uma nova. Nós apenas a atualizamos. Voltando à nossa sigla:

C Create (Criar)
R Read (Ler)
U Update (Atualizar)
D Delete (Apagar)

Hoje o foco é o Update (Atualizar). No Front-End usaremos o método HTTP PUT (ou PATCH). No Back-End, falaremos com os duendes do PostgreSQL para usar a "Bigorna" do banco de dados: o comando UPDATE.

🤓 Dica de Arquitetura (PUT vs PATCH): Em APIs RESTful profissionais, o PUT é usado para substituir o objeto inteiro (se não enviar a quantidade, ela perde-se). Já o PATCH é usado para atualizar apenas uma parte dele (como mudar o material mas manter a quantidade original). Nesta aula simplificamos usando o PUT, mas com o comportamento de uma atualização parcial.

2. A Cláusula de Sobrevivência (WHERE)

O Maior Perigo do SQL

O comando UPDATE serve para alterar os valores das colunas (SET). No entanto, existe uma regra letal que todo programador precisa saber:

-- ⚠️ PERIGO: Se executar isto, TODOS os itens do banco viram Terra!
UPDATE inventario SET material = 'Terra';

-- ✅ CORRETO: Use o WHERE para atualizar apenas UM item específico
UPDATE inventario SET material = 'Espada de Ferro' WHERE id = 12;
O "Thanos Snap" dos Bancos de Dados

Esquecer a cláusula WHERE num comando UPDATE (ou DELETE) é o erro mais catastrófico que um Júnior pode cometer. Sem o WHERE para filtrar o ID do item, o banco aplica a alteração em todas as linhas da tabela de uma só vez, destruindo todos os dados dos utilizadores.

Back-End (Node) 3. A Rota PUT (servidor.js)

Lembrando a Aula 16 (Defesa), não confiamos no Front-End! Validamos se o ID e o Material chegaram vazios. O comando SQL usa $1 e $2 para o WHERE, e incluímos RETURNING * para que a base de dados nos devolva a linha inteira do item atualizado.

Nota Especial: O :id na URL cria um parâmetro dinâmico capturado através de req.params.id. Além disso, adicionamos o parseInt() para garantir que o banco recebe o ID como número inteiro e não como texto!

servidor.js
// Rota PUT para atualizar um item específico (:id é uma variável na URL)
app.put('/api/forjar/:id', async (req, res) => {
    try {
        // 1. Capturamos o ID que veio na URL e o novo material que veio no Body
        const idItem = req.params.id;
        const { novoMaterial } = req.body;

        // VALIDAÇÃO: Defesa contra as Artes das Trevas (Aula 16)
        if (!idItem || !novoMaterial) {
            return res.status(400).json({ erro: 'ID e Novo Material são obrigatórios!' });
        }

        // 2. O Feitiço SQL de Atualização (Com o WHERE protetor!)
        // RETURNING * devolve a linha completa atualizada para enviarmos de volta!
        const sql = 'UPDATE inventario SET material = $1 WHERE id = $2 RETURNING *';
        
        // 3. Executamos na base de dados. Usamos parseInt() como proteção de tipagem!
        const resultado = await pool.query(sql, [novoMaterial, parseInt(idItem)]);

        // Verifica se a base de dados encontrou esse ID
        if (resultado.rowCount === 0) {
            return res.status(404).json({ erro: 'Item não encontrado no mundo.' });
        }

        res.status(200).json({ 
            msg: 'Item reforjado com sucesso!',
            itemAtualizado: resultado.rows[0]
        });

    } catch (erro) {
        console.error(erro);
        res.status(500).json({ erro: 'Falha ao usar a bigorna.' });
    }
});

JS Cliente 4. Fetch PUT e Atualização do DOM

No cliente, anexamos o ID na URL (ex: /api/forjar/12), aplicamos validações visuais sem usar alert(), adicionamos proteção contra cliques duplos e, muito importante: atualizamos a tela (DOM) com o item devolvido pelo RETURNING * do servidor.

cliente.js
const form = document.getElementById('anvil-form');
const areaResultado = document.getElementById('resultado-forja');
const btnSubmit = form.querySelector('button');

form.addEventListener('submit', async (e) => {
    e.preventDefault();
    
    const idItem = document.getElementById('anvil-id').value;
    const novoMaterial = document.getElementById('anvil-material').value.trim();

    // VALIDAÇÃO FRONTAL: Evita enviar lixo para a rede (Feedback direto no DOM)
    if (!idItem || !novoMaterial) {
        areaResultado.innerHTML = '<p style="color: #ef4444;">Preencha os campos da bigorna primeiro!</p>';
        return;
    }

    // PROTEÇÃO DE ESTADO (Loading): Evita múltiplos cliques enviando dezenas de PUTs
    btnSubmit.disabled = true;
    btnSubmit.innerText = 'Forjando...';
    areaResultado.innerHTML = '';

    try {
        // Note que o Método agora é PUT e injetamos o ID dinâmico na URL
        const res = await fetch(`http://localhost:3000/api/forjar/${idItem}`, {
            method: 'PUT',
            headers: { 'Content-Type': 'application/json' },
            body: JSON.stringify({ novoMaterial: novoMaterial })
        });

        const dados = await res.json();
        
        if (!res.ok) throw new Error(dados.erro);
        
        // ATUALIZAÇÃO DO DOM: Reflete a mudança no HTML sem recarregar!
        // O servidor devolveu 'itemAtualizado' por causa do RETURNING *
        areaResultado.innerHTML = `
            <div class="mc-slot mc-slot-result" style="margin: 0 auto;">
                <span style="font-size: 2rem;">✨</span>
            </div>
            <p style="color: #4ade80; margin-top: 5px;">Forjado: ${dados.itemAtualizado.material}</p>
        `;

    } catch (erro) {
        // TRATAMENTO VISUAL DE ERROS (Sem alerts chatos)
        areaResultado.innerHTML = `<p style="color: #ef4444;">Falha na forja: ${erro.message}</p>`;
    } finally {
        // Volta o botão ao normal
        btnSubmit.disabled = false;
        btnSubmit.innerText = 'Forjar (Update)';
    }
});

4. Desafios da Ferraria

01
O Tiro Certeiro (SQL WHERE):

Imagine que quer mudar a quantidade de um bloco de pedra para 64, mas apenas do bloco que tem o ID 5. Escreva no seu caderno a instrução SQL completa (utilizando UPDATE, SET e WHERE).

02
Update Duplo (Node.js):

No código do servidor.js, modifique a query para atualizar não só o material, mas também forçar a coluna quantidade a voltar a ser 1 sempre que um item for reforjado. Dica: A query SQL precisará de um SET material = $1, quantidade = 1.

03
Desafio Épico: O Tratamento de Erro 404

No simulador acima (Seção 3), tente forjar o item com o ID 999 em vez de 12. O que acontece?

No código do servidor (Passo 2), implementámos a lógica if (resultado.rowCount === 0) return res.status(404).... Isso é vital para a operação UPDATE. Ao contrário do SELECT, se tentar fazer UPDATE a um ID que não existe, o banco de dados não dá "erro 500", ele diz "Update bem sucedido em 0 linhas". Como programadores, temos de verificar manualmente o rowCount para avisar o Front-End que o ID inserido não existia no mundo!
