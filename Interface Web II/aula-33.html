Aula 33: O Simbionte Alienígena

Prevenção de XSS (Cross-Site Scripting): Impeça que scripts parasitas infetem o HTML e roubem os dados dos seus utilizadores.

1. O Parasita (A Ameaça do XSS)

Até agora, protegemos o Servidor. Mas existe um ataque que não visa destruir a base de dados, mas sim infetar os utilizadores que visitam o seu site. É o Cross-Site Scripting (XSS).

Imagine que o jornal Clarim Diário tem uma secção de comentários. O Venom (invasor) escreve um comentário, mas em vez de texto, ele injeta um parasita: um bocado de código JavaScript disfarçado de comentário. A Base de Dados guarda isso perfeitamente (ela acha que é só um texto normal).

// O Venom envia isto para o campo de comentários:
<script>
  alert('Nós somos Venom! O teu cookie foi roubado: ' + document.cookie);
</script>

O perigo real acontece a seguir: quando o Homem-Aranha (um utilizador inocente) abre a página para ler as notícias, o site do Clarim Diário descarrega os comentários e "cola-os" no HTML. O navegador do Aranha vê a tag <script> e executa o código malicioso imediatamente. O parasita assumiu o controlo do navegador da vítima!

2. O Contágio (Stored XSS)

A variação mais destrutiva deste ataque chama-se Stored XSS (XSS Armazenado), porque o simbionte fica a viver permanentemente na Base de Dados, aguardando novas vítimas.

1
🕷️
O Invasor Injeta

O Venom posta um comentário contendo <script> no site.

2
💾
O Servidor Guarda

O DB armazena o texto como um comentário normal. Ele não sabe que é um código vivo.

3
😱
A Vítima Acede

Um utilizador abre a página. O HTML carrega o script. O navegador obedece e executa o vírus.

Variação Fantasma: Reflected XSS

Nem todos os parasitas vivem na Base de Dados. No Reflected XSS (XSS Refletido), o código malicioso viaja diretamente na URL (ex: clarim.com/busca?q=<script>...). O servidor recebe o texto de pesquisa, não encontra nada e devolve na página: "Você buscou por: <script>...". Nesse exato momento, o navegador lê o texto refletido e executa o ataque imediatamente, sem nunca gravar o script no servidor!

Simulador de Simbiontes (Clarim Diário)

Veja a diferença entre colar código diretamente no ecrã (Vulnerável) e usar o Escudo Sónico (Seguro).

❌ Sem Proteção (innerHTML)

O site injeta a tag diretamente no ecrã. A imagem quebrada executa o evento "onerror".

Código Executado!
✨ Com Proteção (Escaping / textContent)

O site transforma tags perigosas em texto inofensivo antes de mostrar.

3. O Antídoto Sónico (Escaping HTML)

O simbionte só é perigoso porque o navegador acha que os símbolos de menor < e maior > são partes estruturais da página (tags HTML). A solução para isto chama-se Escaping (ou Codificação de Caracteres).

O Escaping traduz os caracteres especiais para códigos que o navegador apenas desenha no ecrã, mas recusa-se a executar. Por exemplo, < transforma-se na palavra inofensiva &lt; (less than).

❌ No Front-end (Perigo)

Nunca uses innerHTML se o conteúdo vier de um utilizador. Ele converte texto em código vivo.

// 🔴 O ERRO FATAL:
const comentario = "<script>alert(1)</script>";

// Isto infeta a página na hora!
divElement.innerHTML = comentario;

✅ No Front-end (Seguro)

Usa textContent. Ele força o navegador a tratar tudo estritamente como letras, neutralizando as tags.

// 🟢 A SOLUÇÃO:
const comentario = "<script>alert(1)</script>";

// Isto escreve na tela de forma segura:
divElement.textContent = comentario;

Nota: Frameworks modernos como React, Angular e motores de template como EJS fazem este escaping automaticamente por defeito (a menos que os forces a fazer o contrário, como no famigerado dangerouslySetInnerHTML do React).

4. Laboratório Real: Salvando o Clarim Diário

Neste laboratório, criaremos a página de comentários do Clarim Diário. O servidor Node.js receberá os comentários e renderizá-los-á de duas formas: uma falha e uma segura usando uma função de Escaping no back-end.

1 Tabela de Comentários no Neon.tech 

CREATE TABLE comentarios_clarim (
    id SERIAL PRIMARY KEY,
    autor VARCHAR(50),
    conteudo TEXT
);

2 O Servidor e a Interface

Crie o server.js. O próprio servidor enviará o HTML com os comentários para a página. Repare bem na Função Antídoto (escapeHTML).

server.js
const express = require('express');
const { Pool } = require('pg');
const app = express();

app.use(express.urlencoded({ extended: true }));
const pool = new Pool({ connectionString: 'postgres://seu_usuario:senha@neon.tech/neondb' });

// Rota para gravar o comentário na BD (Stored)
app.post('/comentar', async (req, res) => {
    const { autor, conteudo } = req.body;
    // Usamos parâmetros (proteção contra SQLi), mas gravamos o parasita XSS intacto!
    await pool.query('INSERT INTO comentarios_clarim (autor, conteudo) VALUES ($1, $2)', [autor, conteudo]);
    res.redirect('/vulneravel'); // Redireciona para ver o estrago
});

// --- TEMPLATE DO CLARIM DIÁRIO (Visual Jornal) ---
const cabecalho = (titulo) => `
<!DOCTYPE html><html lang="pt-PT"><head><meta charset="UTF-8"><title>Clarim Diário</title>
<style>
    body { font-family: 'Times New Roman', serif; background-color: #f4f4f0; color: #111; max-width: 600px; margin: 0 auto; padding: 20px; }
    .header { text-align: center; border-bottom: 4px double #881337; padding-bottom: 10px; margin-bottom: 20px; }
    .header h1 { font-family: 'Arial Black', sans-serif; text-transform: uppercase; margin: 0; color: #881337; font-size: 36px; letter-spacing: -1px; }
    .header p { font-style: italic; color: #555; margin: 5px 0 0; }
    .comment { background: #fff; padding: 15px; border: 1px solid #ccc; margin-bottom: 10px; box-shadow: 2px 2px 0px rgba(0,0,0,0.1); }
    .comment b { color: #881337; font-family: sans-serif; text-transform: uppercase; font-size: 14px; }
    form { background: #e5e5e5; padding: 15px; border: 1px solid #ccc; margin-top: 20px; }
    input { width: 100%; padding: 8px; margin: 5px 0 15px; box-sizing: border-box; border: 1px solid #999; }
    button { background: #881337; color: white; border: none; padding: 10px 20px; cursor: pointer; font-weight: bold; width: 100%; text-transform: uppercase; }
</style></head><body>
<div class="header"><h1>O Clarim Diário</h1><p>A verdade doa a quem doer!</p></div><h2>${titulo}</h2>`;

const rodapeForm = `
<form action="/comentar" method="POST">
    <b>Deixe a sua opinião:</b><br>
    Autor: <input type="text" name="autor" required>
    Comentário: <input type="text" name="conteudo" required>
    <button type="submit">Enviar</button>
</form></body></html>`;

// 🔴 ROTA VULNERÁVEL: Exibe os comentários construindo o HTML de forma insegura
app.get('/vulneravel', async (req, res) => {
    const resultados = await pool.query('SELECT * FROM comentarios_clarim');
    
    let html = cabecalho('Comentários (Infectados)');
    // PERIGO: As variáveis são concatenadas diretamente no ecrã sem escapar!
    resultados.rows.forEach(c => {
        html += `<div class="comment"><b>${c.autor}</b> disse:<br>${c.conteudo}</div>`; 
    });
    html += rodapeForm;
    
    res.send(html);
});

// --- O ANTÍDOTO SÓNICO ---
// Esta função traduz os caracteres maliciosos para entidades HTML inofensivas
function escapeHTML(str) {
    return str.replace(/[&<>"']/g, function(match) {
        const escapeMap = { '&': '&amp;', '<': '&lt;', '>': '&gt;', '"': '&quot;', "'": '&#39;' };
        return escapeMap[match];
    });
}

// 🟢 ROTA SEGURA: Exibe os comentários aplicando o Antídoto
app.get('/seguro', async (req, res) => {
    const resultados = await pool.query('SELECT * FROM comentarios_clarim');
    
    let html = cabecalho('Comentários (Protegidos)');
    resultados.rows.forEach(c => {
        // DEFESA: Chamamos o escapeHTML() antes de colar no ecrã!
        html += `<div class="comment"><b>${escapeHTML(c.autor)}</b> disse:<br>${escapeHTML(c.conteudo)}</div>`; 
    });
    html += rodapeForm;
    
    res.send(html);
});

app.listen(3000, () => console.log('Clarim Diário online!'));

💣 3. Ataque o Clarim (Rota Vulnerável)

No navegador, aceda a http://localhost:3000/vulneravel. Envie o seguinte comentário:

<script>alert("Foste infectado pelo Venom!")</script>

Resultado: Sempre que qualquer pessoa abrir esta página, o navegador vai ler essa tag que veio do banco de dados e vai executar a caixa de alerta. O ataque XSS Armazenado foi um sucesso.

🛡️ 4. O Escudo Aranha (Rota Segura)

Mude o URL e aceda a http://localhost:3000/seguro.

Resultado: Repare que a caixa de alerta não aparece! Em vez disso, vê o código fonte do script impresso no ecrã de forma inofensiva. A função escapeHTML trocou o < por &lt;, obrigando o browser a tratar o veneno como simples texto decorativo.
